● Bug 找到了，而且很明确：不是菜单项功能没实现，而是整个右键菜单的点击事件被"点
  击空白处关闭菜单"的全局处理器截胡了，所有菜单项都成了摆设。

  根因

  file_tree_context_menu_view（src/main.rs:6310）的菜单容器 div 缺少
  .occlude()。事件链是这样断掉的：

  1. 右键菜单是绝对定位的浮层，渲染在窗口根节点上（src/main.rs:5188），悬浮在
  #main-content-row 上方。
  2. #main-content-row 注册了 on_mouse_down(Left,
  close_menu)（src/main.rs:5019），用于"点击空白处关闭菜单"。
  3. GPUI 里没有 .occlude()
  的元素不会阻挡命中测试。你在菜单项上按下鼠标左键时，事件穿透到下层的
  main-content-row，close_menu（src/main.rs:3682）立即把 file_tree_context_menu
  置为 None 并重绘——菜单在 mouse-down 那一刻就被销毁了。
  4. 菜单项的动作绑定在 on_mouse_up（src/main.rs:6345）。等你松开鼠标时，要么菜
  单元素已经不在渲染树里（handler 不触发），要么即使触发了，handle_file_tree_con
  text_action（src/main.rs:2470）开头的 let Some(target) =
  self.file_tree_context_menu ... else { return; }
  因为菜单状态已被清空而静默返回。

  所以症状就是你观察到的：菜单一点就消失、什么都不发生——对全部 9
  个菜单项都是如此（New File、New Folder、Rename、Delete、Show in File
  Manager、Refresh、Open、Open in New Tab、Filter Files）。

  铁证：同样的 Bug 以前在主菜单栏修过一次

  App 菜单下拉 active_menu_dropdown（src/main.rs:6406）里有一行
  .occlude()，旁边的注释原话就是这个 Bug 的描述：

  ▎ "Without occlude the dropdown does not capture mouse hits, so clicks fall
  ▎ through to the content underneath (which closes the menu) and the item's
  ▎ action is never dispatched."

  当时修主菜单时加了 .occlude()，但文件树右键菜单是后来写的，漏掉了同样的处理。

  功能实现本身的检查结论

  我逐项核对了 handle_file_tree_context_action
  后面的实现，都是真实现，不是空壳：

  - New File / New Folder / Rename → 打开内联命名输入框，Enter 走
  confirm_pending_name 真正创建/重命名（src/main.rs:2645）。
  - Delete → 复用 delete_tree_entry（含之前提交的递归删目录）。
  - Show in System File Manager → Linux 下走
  xdg-open（src/main.rs:5663）。小瑕疵：Linux
  分支只打开父目录、不会选中文件（Windows 的 /select, 和 macOS 的 open -R
  能选中），这是 xdg-open 的能力限制，属合理降级。
  - Refresh / Filter Files / Open / Open in New Tab → 都有完整实现。

  也就是说，修复只需要一处：给 file_tree_context_menu_view 的外层 div 加上
  .occlude()（一行），点击就不会穿透触发 close_menu，on_mouse_up
  能在菜单状态还在时正常派发。要我直接修掉并提交吗？

● 搜索扫描已经门控了。关键看 prepaint 的开头——lines 是不是每帧对全文做 shaping：

  Searched for 1 pattern, read 2 files, ran 1 shell command (ctrl+o to expand)

● 原因基本查清了。Phase 0–4 消掉的是"每键必付"的解析成本，但现在还剩三类大头，而
  且其中两个会被"多个大文档同时打开"放大。

  为什么还卡

  1. 编辑器每次重绘仍对全文做 shaping（最大的剩余瓶颈）

  EditorElement::prepaint（src/main.rs:4568）每帧都调用 shape_text
  对整个文档做字体整形和折行。Phase 4 只解决了 paint
  阶段（跳过屏幕外行）和布局高度缓存，但 prepaint 本身还是 O(全文档)：

  - GPUI 内部有行级布局缓存能吸收重复
  shaping，但它是整个窗口共享、容量有限的缓存。4 个 1.3MB
  文档来回切换标签时会互相挤出缓存，一旦未命中就是几十毫秒级的全文重整形——这正是
  "多开大文档比单开更卡"的直接原因。
  - 即使缓存全命中，每帧仍要收集 3-4 万个 WrappedLine、重建 line_heights、克隆
  line_offsets（src/main.rs:4585-4591），打字和滚动的每一帧都付这笔 O(行数)
  的开销。

  2. 防抖只是移走了解析的时机，解析本身仍在 UI 线程

  1.3MB 文档一次全量解析约 45ms。防抖后它在"停手 80ms"和"连续输入每
  400ms"时触发（src/main.rs:1409），触发的那一帧就是一次肉眼可感的顿挫。防抖减少
  了次数，但没消除单次卡顿。

  3. 多文档的放大效应

  - 每个标签页都缓存整文档的 last_lines（每个字形的几何数据，1.3MB
  文本对应数百万字形，每标签几十 MB 量级），4 个大文档带来明显的内存和 CPU
  缓存压力。
  - 每次按键 shared_document_text 重建（src/main.rs:723），是一次 1.3MB
  的整体拷贝；文档底层是扁平 String，中间插入也是 O(全文) 的
  memmove。单次亚毫秒，但都在热路径上。
  - 自动保存 document.autosave() 在 this.update 里同步写盘（src/main.rs:1470，含
  recovery 副本），每次定时器触发就在 UI 线程写
  1.3MB+，多个脏文档时顿挫频率成倍。

  后续优化方向（按性价比排序）

  1. Phase 5：编辑器可视区虚拟化 shaping——只对视口 ± 过扫描范围内的行做
  shaping（利用已缓存的 line_offsets
  定位可见行，光标/选区几何按需整形所在行）。把编辑器从 O(文档) 变成
  O(视口)，是打字和滚动卡顿的根治方案，也顺带把多标签的缓存互挤问题消掉大半。工
  作量中等偏大，因为光标定位、点击命中、选区绘制目前都依赖全量 lines。
  2. Phase 6：预览解析移到后台线程——用 cx.background_spawn 在后台解析（输入只是
  文本快照），完成后回主线程套用结果。改动小、风险低，直接消掉那个周期性的 ~45ms
  顿挫。可以先做这个。
  3. 自动保存改异步写盘——文本快照丢给后台 executor
  写文件，主线程只更新状态。改动很小。
  4. 零散小优化：非活动标签释放 last_lines（省几十 MB 内存）、prepaint 里
  line_offsets 传 Rc 而不是 clone、line_heights 复用。
  5. 长期方向：文档底层换 rope（如 ropey）+ 增量行索引——让每次编辑从 O(文档)
  降到 O(编辑)，也是未来做真正增量解析的地基。属于大重构，建议放最后。

  我的建议是先做 Phase 6（后台解析，一两小时的活），再做 Phase
  5（编辑器虚拟化，主力工程），这两个分别对应你感受到的"停顿时的顿挫"和"打字/滚
  动/切标签的卡顿"。要开工的话告诉我从哪个开始。

  改动内容：预览解析从"防抖后仍在 UI
  线程同步跑"改为"防抖后丢到后台线程跑，结果回主线程落地"。之前 1.3MB

  45ms，现在这段时间 UI
  完全不受影响，界面继续显示上一版预览，解析完成后自动刷新。

  关键实现点：

  - MarkdownDocument::derive_preview_and_outline(text)（src/lib.rs:1053）：原
  compute_derived 改成不依赖 &self 的关联函数，只吃文本快照，满足 Send
  可跨线程。
  - spawn_preview_parse（src/main.rs:1445）：克隆文本 → cx.background_spawn
  后台解析 → 回主线程把结果套用到标签页并通过 install_derived
  版本门控写回文档缓存（大纲面板继续是缓存命中，不会退化成 UI 线程重新解析）。
  - 竞态防护：每个标签页同时最多一个在途解析；落地时用全局唯一任务 ID
  找回所属标签页（不能用索引——关闭其他标签会使索引移位；也不能用
  text_version——它跨文档可能撞号）。reset_preview_list
  会作废在途任务，替换文档后旧结果直接丢弃。
  - 无空白闪烁：文档的第一次解析仍然同步（打开文件、切到 Split
  模式的首帧），只有后续重解析走后台。
  - 解析期间文档又被编辑的情况：稍旧的结果照常上屏（比冻结好），但版本不匹配会让
  下一帧重新排期解析，install_derived 也会拒绝把旧结果写进缓存。
